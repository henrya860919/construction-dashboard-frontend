# 多專案、多使用者與單／多租戶架構規劃

本文件規劃 Construction Dashboard 的**多專案**、**多使用者**、**單租戶／多租戶**以及**後台分層**，作為前後端實作與資料模型設計的依據。

---

## 一、目標與範圍

| 項目 | 說明 |
|------|------|
| **專案列表** | 拉出為獨立入口頁，使用者先選專案再進入該專案內的儀表板／監測／契約等 |
| **使用者分層** | 區分**系統層**（跨租戶／跨專案）與**專案層**（單一專案內角色） |
| **租戶模式** | 同時考量**單租戶**（一公司一實例）與**多租戶**（一系統多公司／多工地） |
| **後台** | 區分**單租後台**（單一租戶自管）與**多租後台**（平台方／超級管理員） |

---

## 二、名詞定義

| 名詞 | 定義 |
|------|------|
| **租戶 (Tenant)** | 在多租戶模式下，一個獨立客戶／公司／組織單位，資料與權限與其他租戶隔離 |
| **專案 (Project)** | 一個工程專案；在單租戶下可能多專案屬同一公司，在多租戶下專案隸屬於某租戶 |
| **系統層使用者** | 可管理租戶、專案、或跨專案／跨租戶設定的帳號（如平台管理員、租戶管理員） |
| **專案層使用者** | 僅在單一（或數個）專案內有權限的帳號（如專案經理、現場工程師、監造） |
| **單租後台** | 僅管理「當前租戶」內部的專案、使用者、設定（單租戶自管用） |
| **多租後台** | 管理多個租戶、租戶下的專案與使用者、計費或功能開通（平台營運用） |

## 四、專案列表作為獨立入口

### 4.1 流程

1. 使用者登入後，若未帶「當前專案」或需要切換專案，則進入**專案列表頁**。
2. 專案列表頁顯示該使用者有權存取的專案（依角色：系統層可看多租戶／多專案，專案層僅看被授權的專案）。
3. 使用者點選某一專案後，系統記住「當前專案」(current project)，並導向該專案的工作區（例如儀表板或首頁）。
4. 在工作區內，所有 API 與畫面資料皆與「當前專案」綁定；頂欄或側欄可提供**切換專案**回到專案列表或快速切換。

### 4.2 前端路由建議

| 路徑 | 說明 | 備註 |
|------|------|------|
| `/login` | 登入 | 不變 |
| `/projects` | **專案列表**（獨立頁） | 登入後預設進入或從工作區「切換專案」進入 |
| `/projects/:projectId/*` 或 `/p/:projectId/*` | 專案內工作區 | 儀表板、監測、契約等可改為在此前綴下，例如 `/p/:projectId/dashboard` |
| 或維持 `/dashboard`, `/monitoring/...` | 依「當前專案」context 取資料 | 需在 app 層維持 `currentProjectId`（如 Pinia store），API 一律帶 `projectId` |

**兩種常見做法：**

- **A. URL 帶 projectId**  
  - 路由：`/p/:projectId/dashboard`、`/p/:projectId/monitoring/metrics`…  
  - 優點：可書籤、分享連結、瀏覽器前後頁自然對應專案。  
  - 需：layout 與 router 依 `projectId` 載入專案名稱／權限。

- **B. 全域 currentProjectId（不寫進路徑）**  
  - 路由維持現有 `/dashboard`、`/monitoring/...`。  
  - 登入後或從專案列表選完專案後，將 `projectId` 存進 Pinia（或 sessionStorage），所有 API 帶此 id。  
  - 優點：改動小。缺點：分享連結不會綁定專案，需另做「切換專案」流程。

**建議**：若預期會多專案並行、需分享連結，採 **A**；若先求上線、單一專案使用為主，可先採 **B**，之後再改為 A。

### 4.3 側欄與導航

- **專案列表**：可保留在側欄作為獨立項目（例如「專案列表」），點擊後進入 `/projects` 並可再選專案進入工作區。
- 進入專案後，側欄可顯示**當前專案名稱**，並提供「切換專案」回到 `/projects`。
- 概況／監測／契約等維持現有結構，僅在資料層加上「當前專案」篩選。

## 五、使用者分層：系統層 vs 專案層

### 5.1 角色概念（建議）

| 層級 | 角色範例 | 職責範圍 |
|------|----------|----------|
| **系統層** | 平台管理員 (Super Admin)、租戶管理員 (Tenant Admin) | 管理租戶、建立/停用專案、分配專案成員、系統設定（多租時）；單租時可視為「公司管理員」 |
| **專案層** | 專案經理、現場負責人、監造、一般成員 | 僅在一個或多個「被指派」的專案內操作；不能管理租戶或建立專案（依權限） |

### 5.2 權限矩陣（概念）

| 能力 | 系統層（平台/租戶管理員） | 專案層 |
|------|---------------------------|--------|
| 看見／選擇專案列表 | 依範圍：全部專案或本租戶專案 | 僅被授權的專案 |
| 進入專案內工作區 | ✓ | ✓（僅有權的專案） |
| 專案內功能（儀表板、監測、契約…） | ✓ | ✓（可再細分讀寫） |
| 建立/編輯/刪除專案 | 依角色（租戶內或跨租戶） | ✗ |
| 管理專案成員 | ✓（可為） | 依角色（如僅專案經理可加人） |
| 管理租戶（多租時） | 僅平台管理員 | ✗ |
| 後台入口 | 單租後台 / 多租後台 | 通常無或僅單租後台有限功能 |

### 5.3 實作要點

- **後端**：使用者表需能區分「是否為系統層」、所屬租戶（多租時）、以及「專案成員」關聯表（user–project–role）。
- **前端**：登入後取得 `user` 與 `permissions`（或 `roles`），專案列表 API 依此回傳可存取的專案；路由 guard 可依角色決定是否可進後台、是否可進某專案。
- **API**：專案相關 API 需檢查「當前使用者是否為該專案成員或系統層」，避免越權。

## 六、單租戶 vs 多租戶

### 6.1 單租戶 (Single-Tenant)

- **情境**：一間公司／一個組織自用，一套系統只服務自己（或一個主客戶）。
- **資料**：可不設 `tenant_id`，或全表同一 tenant；專案直接隸屬於「這一家」。
- **使用者**：可簡化為「公司管理員 + 專案成員」，不必實作租戶管理介面。
- **後台**：僅需**單租後台**（管理專案、成員、公司設定）。
- **部署**：一實例一 DB 或一實例多專案同一 DB 即可。

### 6.2 多租戶 (Multi-Tenant)

- **情境**：一套系統服務多個公司／多個工地，資料需嚴格隔離。
- **資料**：所有「租戶相關」表都要有 `tenant_id`；查詢時一律帶入當前使用者的 `tenant_id`（或由 JWT / context 取得），避免跨租戶讀寫。
- **使用者**：需區分「平台管理員」（跨租戶）、「租戶管理員」（僅管自己租戶）、「專案層」。
- **後台**：需**多租後台**（平台方）：租戶 CRUD、租戶下的專案與使用者、計費/開通；各租戶可再登入**單租後台**只看自己。
- **部署**：可同一 DB 用 `tenant_id` 隔離，或 DB per tenant（依規模與法規選擇）。

### 6.3 設計時預留多租戶的建議

若未來可能從單租擴成多租，建議：

- **資料表**：核心業務表預留 `tenant_id`（nullable 或 default 單一值），專案表建議有 `tenant_id`（單租時全部同值）。
- **API**：中介層或 middleware 一律從「當前使用者」解析 tenant（單租時回傳固定值），所有查詢/寫入都帶入。
- **前端**：登入後可帶 `tenantId`（單租時可省略或固定），專案列表與專案內 API 皆可依後端權限回傳，前端不需硬編碼租戶邏輯。

## 七、後台：單租後台 vs 多租後台

### 7.1 單租後台（租戶自管）

- **對象**：單一租戶的管理員（或公司管理員）。
- **功能建議**：
  - 專案管理：建立／編輯／停用「本租戶」的專案。
  - 成員管理：邀請／移除使用者、指派專案角色（專案層）。
  - 租戶自身設定：公司名稱、Logo、通知等（不涉及其他租戶）。
- **入口**：可為同一前端的 `/admin` 或子網域，依角色顯示；或獨立子路徑 `/settings`、`/admin`。
- **權限**：僅能操作當前租戶資料；後端所有 API 強制帶入該租戶的 `tenant_id`。

### 7.2 多租後台（平台營運）

- **對象**：平台管理員（Super Admin），可管理所有租戶。
- **功能建議**：
  - 租戶管理：建立／編輯／停用租戶、租戶計費或方案。
  - 專案總覽：可看各租戶下的專案（必要時可代管）。
  - 使用者總覽：可看各租戶的系統層／專案層帳號。
  - 系統設定、功能開關、稽核日誌等。
- **入口**：獨立路徑如 `/platform-admin` 或獨立子網域，僅系統層特定角色可進。
- **權限**：後端依「平台管理員」身份放行跨租戶查詢，但仍需審慎稽核與最小權限。

### 7.3 前後端分工

| 項目 | 單租後台 | 多租後台 |
|------|----------|----------|
| 前端 | 可與主應用同一 SPA，用路由 + 角色控制 | 建議獨立 SPA 或獨立路由區段，避免權限邏輯混雜 |
| 後端 | 同一 API，依 JWT tenant_id 限定範圍 | 專用 API 或同一 API 加 `isPlatformAdmin` 分支 |
| 路由 Guard | 僅允許「租戶管理員」或「專案管理員」等 | 僅允許「平台管理員」 |

## 八、資料模型（概念）

以下為關鍵實體關係，實際欄位依 DB 再細化。

### 8.1 多租戶 + 多專案

```
Tenant (租戶)
  id, name, slug?, plan?, ...

User (使用者)
  id, email, tenant_id?, role_system? (system|tenant_admin|project_user), ...

Project (專案)
  id, tenant_id, name, code?, status, ...

ProjectMember (專案成員)
  id, project_id, user_id, role (project_admin|member|viewer?), ...
```

- 系統層：`User.role_system` 為 platform_admin / tenant_admin；或另做 `UserRole` 表。
- 專案層：僅透過 `ProjectMember` 判斷能否進入某專案及在專案內角色。
- 單租戶時：`Tenant` 可僅一筆，或省略 Tenant 表，`User`/`Project` 無 `tenant_id` 或固定同值。

### 8.2 專案內業務資料

- 監測、契約、概況等表建議帶 `project_id`（與必要時 `tenant_id`）。
- 列表/查詢時一律帶入 `project_id`（及 tenant 若為多租）。

## 九、API 設計要點

1. **認證**：JWT 內建議含 `userId`、`tenantId`（多租）、`currentProjectId`（可選）、`roles` 或 `permissions`。
2. **專案列表**：`GET /api/v1/projects`（或 `/api/v1/users/me/projects`）回傳當前使用者可存取的專案；依系統層/專案層過濾。
3. **專案內 API**：路徑或 query 帶 `projectId`，例如 `GET /api/v1/projects/:projectId/contracts`；後端驗證「使用者是否為該專案成員或系統層」並檢查 tenant。
4. **後台 API**：單租後台僅回傳/寫入本租戶資料；多租後台可帶 `tenantId` 參數或依 admin 身份放行。

## 十、前端實作順序建議

| 階段 | 項目 | 說明 |
|------|------|------|
| 1 | 專案列表獨立頁 | 將 `/projects` 作為登入後預設或主要入口，頁面顯示「可存取的專案」卡片/列表；點選後寫入 `currentProjectId` 並跳轉工作區 |
| 2 | Current Project 狀態 | Pinia store 存 `currentProjectId`、`currentProjectName`；必要時從 API 取得專案列表與當前專案資訊 |
| 3 | 工作區綁定專案 | 所有專案相關 API 呼叫帶入 `currentProjectId`；若採 URL 方案則改為 `/p/:projectId/...` 並從 route 取 projectId |
| 4 | 切換專案 | 頂欄或側欄「切換專案」按鈕 → 回專案列表或彈出專案選擇器，選後更新 store 並重新載入工作區 |
| 5 | 使用者與權限 | 登入回傳 roles/permissions；專案列表與後台入口依權限顯示；路由 guard 阻擋未授權頁面 |
| 6 | 單租後台 | 專案 CRUD、成員管理、租戶自身設定；僅單一租戶資料 |
| 7 | 多租與多租後台 | 若產品確定多租，再引入 tenant_id、多租後台入口與平台管理功能 |

## 十二、總結

| 主題 | 建議 |
|------|------|
| **專案列表** | 拉成獨立入口頁，登入後先選專案再進工作區；側欄保留「專案列表」並支援切換專案 |
| **使用者** | 區分系統層（平台/租戶管理）與專案層（專案成員）；權限與 API 依角色過濾 |
| **單租 vs 多租** | 先單租也可，但資料表與 API 預留 `tenant_id` 與「當前租戶」脈絡，日後擴充多租較省力 |
| **後台** | 單租後台管本租戶專案與成員；多租後台僅平台管理員使用，管所有租戶與系統設定 |
| **實作** | 先做專案列表頁 + currentProject + 專案內 API 帶 projectId，再疊加角色與後台 |

此規劃可隨實際產品決策（先單租、先單一角色等）做階段性裁剪，前後端依同一份文件對齊即可。
